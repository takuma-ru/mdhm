# mdhm Specification

## Purpose

mdhm is a minimal CLI documentation framework for publishing Markdown as static HTML.

- Build static HTML from Markdown with the smallest practical setup.
- Keep customization intentionally minimal.
- Do not add custom Markdown syntax.
- Do not use a configuration file.

## Runtime And Package

- Node.js 24 or later.
- ESM only.
- CLI binary name: `mdhm`.

## Commands

### `mdhm build`

Builds HTML files from Markdown files.

### `mdhm dev`

Starts a development server.

- Live reload is enabled.
- Dev builds are written to `outDir` as real files.
- Default host: `0.0.0.0`.
- Default port: `6348`.

## CLI Options

- `--root-dir <dir>`: input root directory.
- `--out-dir <dir>`: output directory.
- `--host <host>`: dev server host.
- `--port <port>`: dev server port.

Defaults:

- `rootDir`: `.`
- `outDir`: `dist`
- `host`: `0.0.0.0`

No configuration file is supported.

## Markdown

- CommonMark compliant.
- GitHub Flavored Markdown compliant.
- Frontmatter is supported.
- MDX is not supported.
- Custom Markdown syntax is not supported.
- Raw HTML in Markdown is allowed.
- Markdown links are not rewritten.
- Sanitization is not performed in the initial version.
- The implementation should keep a separate sanitization step so sanitization can be added later.

## Output Mapping

Markdown files are converted to HTML files with the same path shape.

- `index.md` -> `index.html`
- `README.md` -> `README.html`
- `foo.md` -> `foo.html`
- `nested/foo.md` -> `nested/foo.html`

Non-Markdown files under `rootDir` are copied to `outDir`.

## Copy And Ignore Rules

The build processes all files under `rootDir`.

- `.md` files are converted to `.html`.
- Non-`.md` files are copied as-is.
- `.gitignore` is respected when present.
- `.git` is always excluded.
- `outDir` is always excluded.
- If `.gitignore` does not exist, everything except `.git` and `outDir` is copied.

Dotfiles are copied unless ignored by `.gitignore` or explicitly excluded above.

## Frontmatter

Frontmatter is supported and may contain arbitrary custom fields.

Special fields:

- `lang`: used for the `<html lang="...">` attribute.
- `head`: used to render tags inside `<head>`.

Example:

```yaml
---
lang: ja
head:
  title: My Page
  description: Page description
---
```

Output behavior:

- `lang` is rendered as `<html lang="ja">`.
- `head.title` is rendered as `<title>`.
- `head.description` is rendered as `<meta name="description">`.
- `head` and `lang` are not rendered into Markdown body content.

## Head Schema

`head` supports the following fields:

- `title`
- `description`
- `meta`
- `link`
- `style`
- `script`

`head.meta`, `head.link`, and object-form `head.script` may use arbitrary keys. Keys are rendered as HTML attributes.

Example:

```yaml
head:
  title: My Page
  description: Page description
  meta:
    - name: author
      content: kcatt
    - property: og:title
      content: My Page
  link:
    - rel: icon
      href: /favicon.svg
  style:
    - |
      :root {
        --mdhm-base: #3b82f6;
      }
  script:
    - src: /app.js
      type: module
      defer: true
    - |
      console.log("inline script");
```

`head.style` supports string and string array values.

`head.script` supports:

- external script objects.
- inline script strings.
- mixed arrays of script objects and inline script strings.

Inline and external scripts are both allowed. mdhm does not guarantee script safety.

## Frontmatter UI

Frontmatter is not mixed into the Markdown body.

If frontmatter exists, mdhm renders a small fixed-position `[frontmatter]` button.

- The button is independent from body content.
- Pressing the button shows the original frontmatter content.
- The displayed format is YAML.
- The button is shown even when frontmatter contains only `head` and `lang`.
- If no frontmatter exists, the button is not shown.

## CSS

mdhm provides minimal built-in CSS.

- Elements do not receive mdhm-specific classes.
- Styles are applied to native HTML elements.
- Dark mode is supported.
- Theme colors are controlled through CSS variables.
- Theme customization has no CLI option.
- Users can override CSS variables through `head.style`.

The CSS delivery method is an implementation detail. It may be inline or external as long as the generated output works.

## Errors

Both `build` and `dev` fail with exit code `1` on:

- invalid frontmatter.
- Markdown render errors.
- HTML generation errors.

In `dev`, these errors also terminate the dev server process.

## Open Discussion Points

These points must be decided before implementation starts.

### 1. Implementation Libraries

Decided libraries:

- Frontmatter parsing: `gray-matter`.
- Markdown and HTML rendering: `unified`, `remark-parse`, `remark-gfm`, `remark-rehype`, `rehype-stringify`.
- `.gitignore` matching: `ignore`.
- File walking: Node.js `fs` APIs.
- CLI argument parsing: `commander`.
- Development server: Node.js `http` server with `sirv` for static file serving.
- File watching: `chokidar`.

Frontmatter is parsed before the Markdown content is passed to unified. The Markdown renderer receives only the frontmatter-stripped Markdown body.

All implementation libraries in this section are decided.

### 2. HTML Document Shape

Generated HTML uses this shape:

```html
<!doctype html>
<html lang="en">
  <head>
    <meta charset="utf-8" />
    <meta name="viewport" content="width=device-width, initial-scale=1" />
    <title>...</title>
    <style>
      /* built-in minimal css */
    </style>
    <!-- head.meta/link/style/script -->
  </head>
  <body>
    <main>
      <article>
        <!-- markdown html -->
      </article>
    </main>
    <!-- frontmatter fixed UI if frontmatter exists -->
    <!-- dev live reload script only in dev -->
  </body>
</html>
```

- Markdown HTML is wrapped in `<main><article>...</article></main>`.
- `<meta charset="utf-8">` is always emitted.
- `<meta name="viewport" content="width=device-width, initial-scale=1">` is always emitted.
- Built-in minimal CSS is emitted in `<head>`.
- `head.meta`, `head.link`, `head.style`, and `head.script` are appended after mdhm defaults.
- Duplicate meta tags are allowed. mdhm does not deduplicate default meta tags and user-provided `head.meta` entries.

### 3. Title Fallback

If `head.title` is missing, the file basename is used as the document title.

Examples:

- `index.md` -> `index`
- `README.md` -> `README`
- `nested/foo.md` -> `foo`

### 4. Frontmatter YAML Display

The `[frontmatter]` panel displays the original frontmatter source string as-is.

### 5. Head Attribute Rendering

Arbitrary attributes in `head.meta`, `head.link`, and object-form `head.script` are rendered with these rules:

- `true` renders a boolean attribute.
- `false`, `null`, and `undefined` omit the attribute.
- String and number values render as escaped attribute values.
- Object and array attribute values are invalid and cause an error.
- Attribute names and values are escaped before output.

### 6. Inline Style And Script Rules

`head.style` supports:

- `string`
- `string[]`

Each string is rendered as a separate `<style>` element.

`head.script` supports:

- `string`
- object
- mixed arrays of strings and objects

Script strings are rendered as inline `<script>` elements.

Script objects are rendered as `<script>` elements with attributes only. Inline content fields such as `content` and `children` are not supported in script objects.

### 7. CSS Variable Schema

Public CSS variables:

- `--mdhm-base`
- `--mdhm-bg`
- `--mdhm-fg`
- `--mdhm-muted`
- `--mdhm-border`
- `--mdhm-surface`
- `--mdhm-link`

Derived colors are based on CSS `color-mix()`.

### 8. Dev Live Reload Mechanism

Live reload uses Server-Sent Events.

- A live reload script is injected only in `dev` output.
- After a successful rebuild, the server sends an SSE message.
- The browser reloads the page when it receives the rebuild message.

### 9. Rebuild Strategy

`dev` performs a full rebuild on every watched file change.

### 10. outDir Cleaning

`outDir` is cleaned before `build` and before each `dev` build.

### 11. Path Safety

Filesystem safety rules:

- `outDir === rootDir` is rejected.
- Empty `outDir` is rejected.
- Filesystem root is rejected as `outDir`.
- Dangerous clean targets are rejected before deleting anything.
